## 二、阶段 1：需求理解

### 2.1 流程

```
产品需求文档 / Jira / 口头描述
        │
        ▼
┌─── 人类 ───────────────────────────┐
│ 1. 阅读需求，形成初步理解            │
│ 2. 标记不清楚的地方                  │
│ 3. 与产品确认关键问题                │
└────────┬───────────────────────────┘
         │
         ▼ 将需求描述交给 AI
┌─── AI ─────────────────────────────┐
│ 1. 提炼核心目标（一句话）             │
│ 2. 识别关键实体和操作                │
│ 3. 列出边界条件和隐含需求             │
│ 4. 生成需求理解确认文档              │
└────────┬───────────────────────────┘
         │
         ▼
┌─── 人类 ───────────────────────────┐
│ 确认 AI 的需求理解是否正确            │
│ 补充遗漏的业务上下文                  │
└────────────────────────────────────┘
```

### 2.2 输入

- 产品需求文档 / Jira Issue / 口头描述
- 相关系统现有文档

### 2.3 输出模板

```markdown
# 需求理解文档

## 基本信息
- **需求来源**：[Jira 链接 / 产品经理]
- **优先级**：P0/P1/P2
- **预计工作量**：[X 人天]

## 需求概述
[用一句话描述核心目标]

## 关键实体
| 实体 | 描述 | 关键属性 |
|------|------|---------|
| [实体名] | [描述] | [属性列表] |

## 用户故事
- 作为 [角色]，我希望 [操作]，以便 [价值]

## 边界条件
- [边界条件 1]
- [边界条件 2]

## 非功能性需求
- 性能：[要求]
- 并发：[要求]
- 数据量：[估算]

## 待确认事项
- [ ] [问题 1]
- [ ] [问题 2]

## 验收标准
- [ ] [标准 1]
- [ ] [标准 2]
```

### 2.4 检查清单

- [ ] 核心目标明确，团队对齐
- [ ] 所有待确认事项已得到答复
- [ ] 验收标准清晰、可测试
- [ ] 已识别对现有系统的影响

## 三、阶段 2：方案设计

### 3.1 流程

```
需求理解文档（已确认）
        │
        ▼
┌─── AI ─────────────────────────────┐
│ 1. 分析受影响的服务和模块             │
│ 2. 设计数据模型变更                  │
│ 3. 设计 API 接口（Proto 定义）       │
│ 4. 生成技术方案文档                  │
│ 5. 生成多个候选方案（如有必要）       │
└────────┬───────────────────────────┘
         │
         ▼
┌─── 人类 ───────────────────────────┐
│ 1. Review 技术方案                   │
│ 2. 评估方案的合理性                  │
│ 3. 考虑长期演进和维护成本             │
│ 4. 决策选择哪个方案                  │
│ 5. 标注修改意见                      │
└────────┬───────────────────────────┘
         │
         ▼
┌─── AI ─────────────────────────────┐
│ 1. 根据反馈修改方案                  │
│ 2. 拆解为子任务                     │
│ 3. 生成测试计划                     │
└────────┬───────────────────────────┘
         │
         ▼
┌─── 人类 ───────────────────────────┐
│ 最终确认：方案 + 任务 + 测试计划      │
└────────────────────────────────────┘
```

### 3.2 方案评审检查清单

| 维度 | 检查项 | 权重 |
|------|--------|------|
| **正确性** | 方案能否正确实现需求？ | 🔴 高 |
| **完整性** | 是否覆盖所有场景（含异常）？ | 🔴 高 |
| **兼容性** | 对现有功能有无影响？ | 🔴 高 |
| **可演进性** | 方案能否支撑未来需求变化？ | 🟡 中 |
| **性能** | 预期负载下能否满足性能要求？ | 🟡 中 |
| **可观测性** | 是否有足够的日志、指标、告警？ | 🟡 中 |
| **可回滚** | 出问题能否快速回滚？ | 🔴 高 |
| **复杂度** | 方案是否过度设计？ | 🟡 中 |

### 4.2 编码规范执行

编码过程中，以下 Rules 自动生效：

| 阶段 | 自动加载的 Rules |
|------|-----------------|
| 写 Go 代码 | `go-code-style.rule` + `readability.rule` + `go-error-handling.rule` |
| 写测试代码 | `go-test.rule` |
| 写 Proto 文件 | `proto-style.rule` |
| 涉及架构变更 | `microservice-design.rule`（手动触发） |

### 4.3 编码过程中的人机交互模式

#### 模式 A：AI 生成 + 人类审查（推荐）

```
人类：描述任务目标和约束
  ↓
AI：生成代码（遵循 Rules）
  ↓
人类：审查 AI 生成的代码
  ↓
人类：修改意见反馈
  ↓
AI：根据反馈修改
  ↓
人类：确认，进入下一任务
```

#### 模式 B：人类编码 + AI 辅助（核心逻辑）

```
人类：手写核心业务逻辑
  ↓
AI：Tab 补全 + 内联建议
  ↓
人类：选择接受/拒绝建议
  ↓
AI：生成对应的测试代码
  ↓
人类：审查测试覆盖度
```

#### 模式 C：AI 全权生成（样板代码/CRUD/工具）

```
人类：描述要生成的样板代码
  ↓
AI：一次性生成完整代码 + 测试
  ↓
人类：快速审查后合入
```

### 4.4 哪些代码适合 AI 生成？

| AI 生成适合度 | 代码类型 | 人类介入程度 |
|:---:|------|:---:|
| ⭐⭐⭐⭐⭐ | CRUD / 样板代码 / 数据结构转换 | 低 |
| ⭐⭐⭐⭐ | 单元测试 / Mock 代码 | 低 |
| ⭐⭐⭐⭐ | Proto 文件 / API 定义 | 中（需确认设计） |
| ⭐⭐⭐ | 工具函数 / 辅助方法 | 中 |
| ⭐⭐⭐ | 配置代码 / CI/CD 脚本 | 中 |
| ⭐⭐ | 核心业务逻辑 | 高（AI 辅助，人主导） |
| ⭐ | 架构代码 / 框架级代码 | 高（人主导） |
| ⭐ | 性能关键路径 | 高（需 Benchmark 验证） |

## 五、阶段 4：测试验证

### 5.1 测试层次

```
┌──────────────────────────────────────┐
│           测试金字塔 + AI 辅助        │
├──────────────────────────────────────┤
│                                      │
│          ┌─────────┐                 │
│          │  E2E    │  ← 人工设计场景  │
│         /│         │\    AI 辅助编写  │
│        / └─────────┘ \               │
│       /   集成测试     \  ← AI 生成   │
│      / ┌─────────────┐ \   人工审查   │
│     /  │ Integration │  \            │
│    /   └─────────────┘   \           │
│   /     单元测试          \  ← AI 生成│
│  / ┌───────────────────────┐\  自动化 │
│ /  │     Unit Tests        │ \       │
│/   └───────────────────────┘  \      │
│                                      │
└──────────────────────────────────────┘
```

### 5.2 自动化检查清单

```bash
# 测试验证流水线

# Step 1: 格式检查
gofmt -l .

# Step 2: 静态分析
go vet ./...

# Step 3: Lint
golangci-lint run ./...

# Step 4: 单元测试 + 竞态检测
go test -race -cover -coverprofile=coverage.out ./...

# Step 5: 覆盖率检查
go tool cover -func=coverage.out | tail -1
# 总覆盖率必须 ≥ 阈值

# Step 6: Proto 规范检查（如有 Proto 变更）
buf lint

# Step 7: Proto 兼容性检查（如有 Proto 变更）
buf breaking --against '.git#branch=main'

# Step 8: 安全漏洞检查
govulncheck ./...
```

### 5.3 测试质量标准

| 指标 | 阈值 | 说明 |
|------|------|------|
| 单测覆盖率（核心逻辑） | ≥ 80% | 核心业务逻辑 |
| 单测覆盖率（工具代码） | ≥ 60% | 辅助工具代码 |
| 测试通过率 | 100% | 不允许 skip 或 ignore |
| Race detection | 0 issues | go test -race 无报警 |
| Lint issues | 0 errors | golangci-lint 无 error |

## 六、阶段 5：审查把关

### 6.1 人机交叉 Review 流程

```
开发者完成编码
        │
        ▼
┌─── 自动化门禁（CI）──────────────────┐
│ ✓ Lint 通过                          │
│ ✓ 测试通过                           │
│ ✓ 覆盖率达标                         │
│ ✓ Proto 检查通过                     │
│ ✓ 安全检查通过                       │
└────────┬─────────────────────────────┘
         │ 门禁全部通过
         ▼
┌─── AI Code Review ───────────────────┐
│ • 规范合规性检查                      │
│ • 常见 Bug 模式识别                   │
│ • 性能隐患扫描                       │
│ • 可读性评分                         │
│ • 生成 Review 报告                   │
└────────┬─────────────────────────────┘
         │ 
         ▼
┌─── 人工 Code Review ────────────────┐
│ • 业务逻辑正确性                     │
│ • 架构合理性                        │
│ • 设计意图是否与方案一致              │
│ • AI Review 报告中标记的问题确认      │
│ • 最终 Approve / Request Changes     │
└────────┬────────────────────────────┘
         │
         ▼
     合入主干
```

### 6.2 Review 关注点分层

| 层次 | 关注点 | 负责方 | 自动化程度 |
|------|--------|--------|-----------|
| L0 格式 | 代码格式、Import 排序 | CI 自动 | 100% |
| L1 规范 | 命名、注释、错误处理 | AI Review | 90% |
| L2 安全 | SQL 注入、硬编码密钥 | AI + 人工 | 70% |
| L3 性能 | 不必要的分配、缺失 timeout | AI + 人工 | 50% |
| L4 逻辑 | 业务逻辑正确性 | 人工为主 | 10% |
| L5 架构 | 设计合理性、可演进性 | 人工 | 0% |

### 6.3 Review 效率优化

| 优化策略 | 说明 |
|---------|------|
| **小批量 PR** | 每个 PR 不超过 400 行变更 |
| **描述清晰** | PR 描述包含：目的、方案概要、影响范围 |
| **自测先行** | 提交 Review 前确保自测通过 |
| **AI 预审** | 先让 AI Review，过滤低级问题 |
| **聚焦高价值** | 人工 Review 聚焦 L4/L5，低层级交给自动化 |

## 八、Debug 场景的人机协作

> 你在原文中提到了一个关键问题：代码越来越多是 AI 写的，人类越来越不熟悉，出了线上问题怎么办？

### 8.1 Debug SOP

```
线上告警/用户反馈
        │
        ▼
┌─── 人类 ──────────────────────────┐
│ 1. 评估影响面和严重程度             │
│ 2. 收集现场信息（日志、指标、请求）  │
└────────┬──────────────────────────┘
         │ 将信息交给 AI
         ▼
┌─── AI ────────────────────────────┐
│ 1. 分析日志和错误信息               │
│ 2. 关联代码变更（git log/blame）    │
│ 3. 定位可能的问题代码               │
│ 4. 提出修复方案（含影响分析）       │
└────────┬──────────────────────────┘
         │
         ▼
┌─── 人类 ──────────────────────────┐
│ 1. 确认 AI 的分析是否靠谱           │
│ 2. 决策：修复 or 回滚               │
│ 3. 审批修复方案                     │
└────────┬──────────────────────────┘
         │
         ▼
┌─── AI ────────────────────────────┐
│ 1. 实施修复（编码）                 │
│ 2. 编写回归测试                    │
│ 3. 本地验证                       │
└────────┬──────────────────────────┘
         │
         ▼
┌─── 人类 ──────────────────────────┐
│ 1. 审查修复代码                    │
│ 2. 决策上线                       │
│ 3. 上线后观察                     │
│ 4. 编写事后复盘（AI 辅助）         │
└───────────────────────────────────┘
```

### 8.2 关键洞察

> **你不需要记住 AI 写的每一行代码，但你需要理解系统的架构和数据流。**

| 层次 | 人类需要了解 | AI 可以做 |
|------|------------|----------|
| 架构层 | 服务间的关系、数据流向 | 生成架构图、调用关系 |
| 模块层 | 每个服务的核心职责 | 生成模块文档、API 文档 |
| 代码层 | 不需要逐行了解 | 代码检索、定位、解释 |
| 数据层 | 核心数据模型 | 生成 ER 图、数据字典 |
