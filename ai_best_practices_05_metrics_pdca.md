### 1.2 核心指标定义

#### A. 效率指标

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **需求交付周期** | 从需求确认到上线的日历天数 | Jira 状态时间差 | 缩短 30% |
| **编码效率** | 每人每天有效代码行数（含 AI 辅助） | Git 统计 | 提升 50% |
| **Review 周转时间** | 从提交 PR 到 Approve 的时间 | Git 统计 | ≤ 4 小时 |
| **AI 代码采纳率** | AI 建议被接受的比例 | IDE 统计 | > 60% |
| **首次 Review 通过率** | 无需修改即通过 Review 的 PR 比例 | Git 统计 | > 70% |

#### B. 质量指标

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **缺陷密度** | 每千行代码的 Bug 数 | Bug数 / KLOC | 降低 40% |
| **测试覆盖率** | 代码被测试覆盖的比例 | go test -cover | ≥ 70% (总体) |
| **Lint 问题密度** | 每千行代码的 lint 问题数 | golangci-lint / KLOC | ≤ 1 |
| **Proto 规范合规率** | Proto 文件通过 buf lint 的比例 | buf lint | ≥ 95% |
| **线上故障率** | 每次发布引入的线上故障数 | 监控统计 | ≤ 0.1 次/发布 |
| **MTTR** | 平均故障修复时间 | 监控统计 | ≤ 30 分钟 |

#### C. AI 协作指标

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **AI 使用频率** | 每人每天 AI 辅助编码的次数 | IDE 统计 | > 20 次/天 |
| **AI 覆盖率** | AI 辅助生成的代码占比 | 估算 | Phase1: 30%, Phase3: 60% |
| **Rules 有效性** | Rules 触发后 AI 遵循的比例 | 抽样检查 | > 90% |
| **Skills 使用频率** | 每周 Skills 的调用次数 | 统计 | > 5 次/周/人 |
| **Rules 贡献数** | 团队成员自主贡献的 Rules 数量 | Git 统计 | Phase3: ≥ 20 条 |

#### D. 团队成长指标

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **工程素养评分** | 季度评估团队工程素养水平 | 测验 + 代码审查 | 季度环比提升 |
| **新人 Onboarding 时间** | 新人从入职到独立交付的天数 | 跟踪记录 | 缩短 50% |
| **知识分享频次** | 每月技术分享/文章数 | 统计 | ≥ 2 次/月 |
| **培训满意度** | 培训后问卷评分 | 问卷 | ≥ 4.0/5.0 |

### 2.2 数据采集脚本示例

```bash
#!/bin/bash
# metrics.sh - 项目质量度量采集

echo "📊 项目质量度量报告"
echo "===================="
echo "日期: $(date +%Y-%m-%d)"
echo ""

# 1. 代码量统计
echo "## 1. 代码量"
echo "Go 代码行数: $(find . -name '*.go' ! -name '*_test.go' | xargs wc -l | tail -1 | awk '{print $1}')"
echo "测试代码行数: $(find . -name '*_test.go' | xargs wc -l | tail -1 | awk '{print $1}')"
echo "Proto 文件数: $(find . -name '*.proto' | wc -l)"
echo ""

# 2. 测试覆盖率
echo "## 2. 测试覆盖率"
go test -coverprofile=coverage.out ./... 2>/dev/null
COVERAGE=$(go tool cover -func=coverage.out | tail -1 | awk '{print $3}')
echo "总体覆盖率: $COVERAGE"
echo ""

# 3. Lint 问题
echo "## 3. Lint 问题"
LINT_ISSUES=$(golangci-lint run ./... 2>&1 | grep -c ":")
echo "Lint 问题数: $LINT_ISSUES"
KLOC=$(find . -name '*.go' | xargs wc -l | tail -1 | awk '{printf "%.1f", $1/1000}')
echo "问题密度: $(echo "scale=2; $LINT_ISSUES / $KLOC" | bc) / KLOC"
echo ""

# 4. Proto 规范
echo "## 4. Proto 规范"
if command -v buf &> /dev/null; then
    PROTO_ISSUES=$(buf lint 2>&1 | grep -c ":")
    echo "Proto lint 问题: $PROTO_ISSUES"
else
    echo "buf 未安装，跳过"
fi
echo ""

# 5. 安全检查
echo "## 5. 安全检查"
if command -v govulncheck &> /dev/null; then
    VULN_COUNT=$(govulncheck ./... 2>&1 | grep -c "Vulnerability")
    echo "已知漏洞: $VULN_COUNT"
else
    echo "govulncheck 未安装，跳过"
fi

# 清理
rm -f coverage.out
```

### 2.3 Git Hook 采集

```bash
#!/bin/bash
# .git/hooks/pre-push - 推送前自动采集数据

# 记录本次推送的统计信息
TIMESTAMP=$(date +%Y-%m-%d_%H:%M:%S)
AUTHOR=$(git config user.name)
CHANGES=$(git diff --stat HEAD~1 | tail -1)

echo "$TIMESTAMP | $AUTHOR | $CHANGES" >> .metrics/push_log.csv
```

## 四、PDCA 循环设计

### 4.1 PDCA 节奏

```
       ┌─────── 月度 PDCA ──────────────────────────────────────┐
       │                                                         │
       │   ┌─── 周 PDCA ───┐  ┌─── 周 PDCA ───┐               │
       │   │                │  │                │               │
       │   │  P → D → C → A │  │  P → D → C → A │  ...         │
       │   │                │  │                │               │
       │   └────────────────┘  └────────────────┘               │
       │                                                         │
       │   Plan: 周一晨会确认本周重点                              │
       │   Do:   周二～周四执行                                    │
       │   Check: 周五回顾度量数据                                 │
       │   Act:  调整下周计划                                      │
       │                                                         │
       └─── 月末：月度回顾，调整下月策略 ──────────────────────────┘
       
       ┌─────── 季度 PDCA ──────────────────────────────────────┐
       │                                                         │
       │   月度 PDCA  →  月度 PDCA  →  月度 PDCA                 │
       │                                                         │
       │   季度末：深度复盘 + ROI 分析 + 调整年度策略              │
       │                                                         │
       └────────────────────────────────────────────────────────┘
```

### 4.2 每个阶段的 PDCA

#### Phase 1 (M1-M3) — 筑基期

| 环节 | 内容 |
|------|------|
| **Plan** | 目标：Rules 体系建立 + 基础培训完成 + 代码质量基线改善 |
| **Do** | 执行：每周培训 + Rules 逐步上线 + Commands 实战 |
| **Check** | 度量：lint 问题密度、覆盖率、培训满意度、AI 使用频率 |
| **Act** | 调整：优化不好用的 Rules、补充缺失的 Rules、调整培训节奏 |

#### Phase 2 (M4-M6) — 深耕期

| 环节 | 内容 |
|------|------|
| **Plan** | 目标：完整 SOP 运转 + Skills 覆盖核心场景 + 质量门禁上线 |
| **Do** | 执行：SOP 逐步推广 + Skills 开发 + CI 集成 |
| **Check** | 度量：交付周期、首次 Review 通过率、缺陷密度、Skills 使用频率 |
| **Act** | 调整：优化 SOP 中的卡点、改进 Skills 准确度 |

#### Phase 3 (M7-M9) — 飞轮期

| 环节 | 内容 |
|------|------|
| **Plan** | 目标：团队自驱 + 知识库自演化 + ROI 证明 |
| **Do** | 执行：团队自主贡献 Rules + 全流程运转 + 度量优化 |
| **Check** | 度量：全维度指标对比、ROI 计算、团队满意度 |
| **Act** | 调整：制定下阶段策略（深化/扩展/新方向） |

### 5.3 应对"只看需求迭代效率"的管理风格

这是你提到的核心痛点。应对策略：

| 策略 | 具体做法 |
|------|---------|
| **用效率数据说话** | 展示：AI 辅助后需求交付速度提升了 X% |
| **用质量换算效率** | 展示：减少 X 个 Bug = 节省 X 天返工时间 = 多做 X 个需求 |
| **用事故成本对比** | 展示：上次线上事故耗费 X 人天修复，如果有质量门禁可以避免 |
| **渐进式推进** | 不要一口气推太多，每次只增加 1-2 个改进点，让效果说话 |
| **借力打力** | 用"AI 自动检查"替代"人工强制审查"，降低推行阻力 |
