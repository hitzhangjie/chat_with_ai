你的理解非常精准。**大模型（Weights/Model）本身只是一个黑盒函数**，它接收的是特定格式的 Tensor（张量）。而 **API Provider（服务提供商）** 则是在模型之上加了一层“翻译官”或“适配层”，负责将 HTTP 请求（如 JSON 格式）转换为模型能理解的输入。

所谓“兼容 OpenAI 协议”，本质上就是这些提供商在自己的服务器前端实现了一套和 OpenAI 完全一致的接口路径（如 `/v1/chat/completions`）和参数格式。

---

## 哪些服务商兼容 OpenAI 协议？

目前，OpenAI 的接口规范已经成为了大模型行业的“事实标准”。为了降低开发者的迁移成本，绝大多数提供商都选择了主动兼容。

### 1. 原生/高度兼容的服务商
这些平台提供的 API 几乎可以无缝替换 OpenAI 的 SDK，只需更改 `base_url` 和 `api_key`：

| 服务类别 | 代表厂商 | 特点 |
| :--- | :--- | :--- |
| **主流商业闭源** | **DeepSeek (深度求索)**, **Groq**, **Together AI**, **Mistral AI** | 几乎 100% 参照 OpenAI 协议设计，支持流式输出和 Tool Call。 |
| **国内大厂** | **月之暗面 (Kimi)**, **智谱 AI (GLM)**, **通义千问 (DashScope)** | 提供专门的 OpenAI 兼容端点，方便国内开发者接入。 |
| **云服务集成商** | **Microsoft Azure OpenAI**, **OpenRouter** | Azure 是官方合作伙伴自不必说；OpenRouter 则是聚合了数百个模型并统一成 OpenAI 格式。 |
| **本地部署工具** | **Ollama**, **vLLM**, **LocalAI**, **LM Studio** | 这些推理框架启动后，默认就会在本地开启一个兼容 OpenAI 协议的 HTTP 服务。 |

### 2. 不（完全）兼容的服务商
有些厂商拥有深厚的生态护城河，或者有独特的交互逻辑，因此坚持使用自己的协议：

* **Google Gemini (Google AI Studio/Vertex AI):** 使用自己的 SDK 格式（如 `generateContent`），参数结构与 OpenAI 差异较大。
* **Anthropic (Claude):** 拥有独立的 API 协议（如 `messages` API），虽然逻辑相似，但字段名称不同（例如使用 `max_tokens` 限制时，其位置和结构有别）。
* **AWS Bedrock:** 作为一个云平台，它通过 AWS 自己的签名机制和输入输出格式来调用各种模型。

---

## 为什么有些 API 不兼容？

既然兼容 OpenAI 能带来更多用户，为什么还有厂商“不合群”？

1.  **原生功能差异：** 有些模型支持特殊能力。例如，Google Gemini 支持超长上下文（200万+ tokens）和多模态原生输入（直接传视频文件），OpenAI 的协议框架有时难以完美承载这些扩展功能。
2.  **生态护城河：** 像 Google 或 Anthropic 这样的巨头，希望开发者深度绑定在自己的 SDK 环境中，从而提高迁移成本。
3.  **技术演进：** 在 OpenAI 协议完善之前，很多厂商已经开发了自己的接口。大规模改动既有接口会影响存量用户。

## 参数是如何转换的？（技术内幕）

正如你所感知的，API 提供商做了一层逻辑转换。我们以一个 `chat` 请求为例：

1.  **用户输入：** 你发送一个包含 `messages` 数组的 JSON。
2.  **API 适配层：** 提供商的后端（通常用 Go 或 Python 编写）解析这个 JSON，并根据所选模型加载对应的 **Prompt Template（提示词模板）**。
3.  **转换逻辑：**
    * 如果是 Llama 3 模型，它会将 `messages` 转换为 `<|begin_of_text|><|start_header_id|>user<|end_header_id|>...` 这种格式。
    * 如果是代码模型，它可能需要加入特定的 Stop Tokens。
4.  **推理引擎：** 最终转换后的纯文本（字符串）被 Tokenize 变成数字序列，输入到大模型的权重矩阵中进行计算。
5.  **结果包装：** 模型生成的下一个 Token 被转回文字，再包装成 OpenAI 风格的 `choices` 格式返回给你。
