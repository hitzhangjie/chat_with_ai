# 深度解构 AI Agent：规划（Planning）的实现逻辑与思维演进

## 引言
在 AI Agent 的公式中：**Agent = LLM + Planning + Memory + Tools**。
很多人最初的理解是：Planning 仅仅是让 LLM 多输出几行文字。但在实际的工业级开发（尤其是使用 Go 或 LangChain/LangGraph 框架）中，Planning 远比“自回归输出”要复杂得多。本文将带你重新认识 Agent 的“大脑”是如何进行任务编排与决策的。

---

## 一、 重新定义 Planning：它不只是 LLM 的输出

在 ReAct 推理循环中，Planning 的本质是**将高阶目标（High-level goal）分解为可执行的原子任务（Atomic tasks）**。

### 1. 核心工作内容
* **任务分解 (Task Decomposition)**：利用 CoT（思维链）等技术，将复杂问题拆解为 Step-by-step 的子任务。
* **自我反思与修正 (Self-Reflection)**：在执行过程中观察结果，如果发现偏离目标，动态调整后续计划。

### 2. 实现逻辑的两种路径
在工程实现上，我们不能完全依赖 LLM 的随机性，通常结合以下两种方式：
* **内在规划（模型驱动）**：完全依靠 Prompt 工程，让模型输出 JSON 格式的任务列表。优点是灵活，缺点是长路径下容易产生“幻觉”或逻辑漂移。
* **外部约束规划（架构驱动）**：通过 **状态机（State Machines）** 或 **DAG 图** 约束。LLM 只负责在预设的节点中做“选择题”，而逻辑跳转由外部代码（如 Go 框架）控制，这更加稳定可靠。



---

## 二、 核心算法对撞：CoT vs. ToT

对于初学者来说，理解 Planning 最关键的是分清两种主流的思维模式：**思维链（CoT）** 与 **思维树（ToT）**。

### 1. CoT (Chain of Thought) —— 线性的“直觉推理”
* **逻辑**：A → B → C → 结果。
* **隐喻**：像小学生做数学题，必须写出步骤。
* **痛点**：它是“走一步看一步”，一旦中间某一步逻辑出错，整个任务就会彻底失败，没有纠错机制。

### 2. ToT (Tree of Thoughts) —— 发散的“全局搜索”
* **逻辑**：在每一步推理时，生成多个候选方案，并通过一个“评价器”打分，选择最优路径。
* **隐喻**：像高手下围棋，走一步看三步，甚至会“悔棋”（回溯）去尝试另一种可能。
* **优势**：具备极强的容错性和全局最优解搜索能力，适合复杂的算法设计或创意策划。



### 3. 快速对比表

| 特性 | CoT (思维链) | ToT (思维树) |
| :--- | :--- | :--- |
| **思考形状** | 直线 (Linear) | 树状 (Tree) |
| **容错性** | 低（一步错步步错） | 高（可回溯、剪枝） |
| **计算成本** | 低（单次推理） | 高（多次采样与评估） |
| **适用场景** | 确定性强的简单逻辑 | 复杂决策、多方案对比 |

---

## 三、 总结：从“碎碎念”到“状态管理”

从小白到专家的进阶，在于意识到 Planning 不再是 LLM 的“碎碎念”，而是一套**结构化的状态管理**。

在构建 Agent 时：
1.  如果任务简单，**CoT** 足矣；
2.  如果任务复杂且容错率低，需要引入 **ToT** 甚至 **Graph（图）** 架构；
3.  最稳健的做法是**用代码（如 Go 的条件判断、状态机）来约束 LLM 的规划行为**，而非让它盲目生成。
