## 二、软件工程方法论

### 2.1 DORA 指标（DevOps Research and Assessment）

DORA 四大指标是衡量软件交付效能的黄金标准：

| 指标 | 定义 | Elite 水平 | 与本方案的关系 |
|------|------|-----------|-------------|
| **部署频率** | 多久部署一次到生产 | 按需（一天多次） | Phase 2: CI/CD 自动化 |
| **变更前置时间** | 从代码提交到生产运行 | < 1 小时 | Phase 1-2: 流程优化 |
| **变更失败率** | 部署导致生产故障的百分比 | 0-15% | Phase 2: 质量门禁 |
| **MTTR** | 从故障到恢复的时间 | < 1 小时 | Phase 3: Debug AI 辅助 |

**应用方式**：
- Phase 0 采集基线数据
- 每月对比 DORA 四指标趋势
- 以 Elite 水平为长期目标

### 2.2 持续交付（Continuous Delivery）

> 来源：Jez Humble & David Farley《持续交付》，乔梁《持续交付 3.0》

核心原则：

| 原则 | 说明 | 本方案映射 |
|------|------|----------|
| **构建质量内建** | 不靠后期检查，在流程中内建质量 | Rules 自动约束 + 质量门禁 |
| **小批量交付** | 小步快跑，降低风险 | 任务拆解 ≤ 4h/个 |
| **自动化一切** | 能自动化的绝不手动 | CI/CD + AI 自动检查 |
| **持续反馈** | 快速获得反馈，快速调整 | PDCA + 度量体系 |
| **人人有责** | 质量不是 QA 的事 | 全员参与 Rules 贡献 |

**乔梁《持续交付 3.0》的关键增量**：
- **双环模型**：业务探索环 + 技术能力环
- **反馈驱动**：每个环节都有反馈，形成闭环
- **全局优化**：不只优化开发效率，优化全价值流

### 2.3 TDD / BDD

| 方法 | 核心理念 | 本方案应用 |
|------|---------|----------|
| **TDD** | 先写测试 → 写实现 → 重构 | `tdd-workflow.skill` |
| **BDD** | 用业务语言描述行为 → 自动化验证 | 需求理解阶段的验收标准 |

### 2.4 领域驱动设计（DDD）

| DDD 概念 | 说明 | 本方案应用 |
|----------|------|----------|
| **限界上下文** | 明确的业务边界 | 微服务边界划分 |
| **通用语言** | 团队统一的业务术语 | Rules 中的术语定义 |
| **聚合根** | 一致性边界 | 存储设计规范 |
| **反腐层** | 隔离外部系统变化 | 架构设计 Rule |

### 2.6 The Twelve-Factor App

与 Go 微服务相关的关键因素：

| # | 因素 | Go 微服务最佳实践 |
|---|------|------------------|
| III | **Config** | 环境变量 / 配置中心，不硬编码 |
| IV | **Backing Services** | DB/MQ/Cache 作为可替换的附加资源 |

| IX | **Disposability** | 快速启停，优雅关闭 |
| XI | **Logs** | 日志作为事件流，stdout 输出 |
| XII | **Admin** | 管理任务作为一次性进程 |

### 3.3 PDCA 循环（戴明环）

> 已在 05 文档中详细展开。

核心理念：**管理是一个科学实验过程**。

```
    Plan（计划）                    Do（执行）
    基于数据制定改进计划              执行计划，收集数据
         │                              │
         │      ┌──────────┐            │
         │      │          │            │
         └─────▶│  PDCA    │◀───────────┘
                │  循环    │
         ┌─────▶│          │◀───────────┐
         │      └──────────┘            │
         │                              │
    Act（处置）                   Check（检查）
    标准化成功做法，               对比目标和实际，
    纠正问题，                    分析偏差根因
    进入下一个循环
```

### 3.4 5 WHY 分析法

用于根因分析，在 Check 阶段使用：

```
问题：测试覆盖率一直达不到 70%
  ↓ WHY?
因为：开发者不写测试
  ↓ WHY?
因为：写测试太耗时
  ↓ WHY?  
因为：代码可测试性差，Mock 困难
  ↓ WHY?
因为：没有使用依赖注入，直接依赖全局变量
  ↓ WHY?
因为：缺乏可测试性设计的规范和培训

→ 根因：需要在 Rules 中增加可测试性设计规范 + 培训
```

### 4.1 Harness Engineering

> 2025-2026 年的新兴概念，替代 Prompt Engineering

**核心理念**：不只是写好提示词，而是系统性地**驾驭** AI 的完整行为。

```
│   └─────────────────────────────────────┘      │
│                                                 │
└────────────────────────────────────────────────┘
```

### 4.2 Context Engineering

> Prompt Engineering 的进化：不只关注提示词，更关注如何提供完整、高质量的上下文。

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|-------------------|
| 焦点 | 单条提示词的措辞 | 整体上下文的组织 |
| 范围 | 单次对话 | 跨会话、跨项目 |
| 信息来源 | 手动提供 | 自动采集（MCP、RAG） |
| 知识 | 临时注入 | 持久化（Rules、Memory） |
| 目标 | 获得好的单次回答 | 建立稳定的 AI 行为模式 |

**在本方案中的体现**：
- Rules = 预加载的上下文（工程规范）
- Skills = 结构化的操作上下文（工作流）
- MCP = 动态获取的项目上下文（数据库、文档）
- Memory = 跨会话积累的上下文（学习记录）

### 4.3 Agentic Workflow

> AI 从"回答问题"到"自主执行复杂任务"的演进。

```
Level 0: 补全模式
  人类写代码 → AI 补全当前行
  
Level 1: 对话模式
  人类描述需求 → AI 生成代码片段 → 人类手动应用
  
Level 2: 编辑模式
  人类描述需求 → AI 直接修改文件 → 人类审核
  
Level 3: Agent 模式  ← 当前主流
  人类描述任务 → AI 规划步骤 → AI 执行多步操作 → 人类审核
  
Level 4: Multi-Agent 模式  ← 前沿探索
  人类描述目标 → 多个 AI Agent 协作 → 人类最终审核
  
Level 5: Autonomous 模式  ← 未来（Vibe Coding）
  人类描述愿景 → AI 自主完成 → 人类验收
```

**本方案定位**：Phase 1-2 在 Level 2-3，Phase 3 探索 Level 3-4。

### 4.4 RAG（Retrieval-Augmented Generation）

```
用户提问 → 检索相关文档 → 将文档作为上下文 → AI 基于上下文生成回答

在本方案中：
  Rules 文件 ≈ 静态 RAG（每次对话自动注入）
  MCP 连接 ≈ 动态 RAG（按需检索项目知识）
  Memory ≈ 个性化 RAG（基于历史积累的上下文）
```

### 7.2 关于你提到的"拔苗助长"的思考

你说"感觉有点拔苗助长"，这是一个非常真实的担忧。我的分析：

**不是拔苗助长，而是"修渠引水"**：

```
传统方式（拔苗助长）：
  强制要求 → 初级工程师达不到 → 挫败感 → 放弃

本方案（修渠引水）：
  编写 Rules → AI 自动执行规范 → 初级工程师在使用中学习
                                   ↓
                           潜移默化，水到渠成
```

**关键区别**：
1. 不是要求初级工程师**一夜之间**掌握所有技能
2. 而是通过 AI 让规范**自动生效**，同时初级工程师在实践中**逐渐理解**为什么
3. 就像使用 `gofmt` —— 你不需要先学会所有格式规则，工具自动帮你做
4. 但用久了，你自然就知道了 Go 的代码风格

### 7.3 关于"将来能否 100% Vibe Coding"

你的判断非常理性：

> "核心系统、核心服务里面写这些，断无可能。"

我的补充分析：

| 场景 | Vibe Coding 适用度 | 原因 |
|------|:---:|------|
| 内部工具/脚本 | ⭐⭐⭐⭐⭐ | 影响面小，快速迭代 |
| 管理后台 | ⭐⭐⭐⭐ | 相对标准化，CRUD 为主 |
| 非核心微服务 | ⭐⭐⭐ | 可以大量 AI 辅助，但需 Review |
| 核心业务服务 | ⭐⭐ | AI 辅助编码，人主导设计和审查 |
| 核心系统框架 | ⭐ | 人类主导，AI 仅辅助 |
| 金融/安全相关 | ⭐ | 必须人类深度参与 |

**结论**：Vibe Coding 不是目标，**可靠高效交付**才是目标。AI 是手段，不是目的。
