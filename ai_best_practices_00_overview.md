# AI 时代程序员自我修养 — 总纲与实施蓝图

> "给 AI 立规矩的人，才是 AI 时代最有价值的人。" 

## 一、问题全景

| 维度 | 现状痛点 | 期望目标 |
|------|---------|---------|
| **工程素养** | 初级工程师基础薄弱，老码农经验碎片化，缺乏系统性工程规范 | AI 作为"随身教练"，在日常编码中潜移默化传递工程素养 |
| **AI 工具应用** | 停留在"提示词 + 代码"的初级阶段 | 建立完整的人机协作工作流，沉淀可复用的 Rules/Skills/Commands |
| **交付质量** | 代码量膨胀、服务难维护、API/PB 协议混乱 | 在提效的同时保证可靠交付，质量可量化、可追踪 |
| **团队成长** | 只看需求迭代效率，忽视工程能力建设 | 通过 AI 辅助实现"做中学"，渐进提升团队整体工程水平 |

### Phase 0：准备期（2 周）

**目标**：完成工具选型、环境搭建、基线度量

| 周次 | 交付物 | 负责人 |
|------|--------|--------|
| W1 | AI IDE 选型报告（参见 02 文档）；试点团队确定（2-3人先行） | Tech Lead |
| W1 | 现有代码质量基线报告（lint 问题数、测试覆盖率、PB 规范符合度） | 全员 |
| W2 | 开发环境统一配置（AI IDE + 插件 + 基础 Rules 文件） | Tech Lead |
| W2 | 首批 5 条核心 Rules 编写完成（参见 03 文档） | Tech Lead |

**验收标准**：
- [ ] 全员完成 AI IDE 安装配置
- [ ] 基线度量数据收集完毕
- [ ] 首批 Rules 通过团队评审

---

### Phase 1：筑基期（Month 1 - Month 3）

**目标**：工程素养培训 + 基础人机协作流程建立

#### Month 1 — Go 编码规范与基础工具链

| 周次 | 培训主题 | AI 协作落地 | 交付物 |
|------|---------|------------|--------|
| W1-W2 | Go 编码规范（命名、包结构、错误处理） | 编写 `go-code-style.rule` | Rule 文件 + 培训材料 |
| W3 | Go 工具链（go vet / golangci-lint） | 编写 `lint-check.command` | Command 文件 + CI 集成 |
| W4 | Go 安全检查（go vulncheck） | 编写 `security-scan.command` | Command 文件 |

#### Month 2 — 测试素养与 API 规范

| 周次 | 培训主题 | AI 协作落地 | 交付物 |
|------|---------|------------|--------|
| W5-W6 | 单元测试 / Mock 测试 / Table-driven tests | 编写 `go-test.rule` + `gen-test.command` | Rule + Command |
| W7 | Race detection / Fuzz testing / Benchmark | 编写 `race-fuzz-bench.command` | Command 文件 |
| W8 | PB 规范 / Google API 设计指南 / ProtoCheck | 编写 `proto-style.rule` | Rule + Uber protocheck 集成 |

#### Month 3 — Code Review 人机协作

| 周次 | 培训主题 | AI 协作落地 | 交付物 |
|------|---------|------------|--------|
| W9-W10 | 可读性（Readability）/ 可测试性 | 编写 `readability.rule` | Rule 文件 |
| W11 | Code Review SOP（人机交叉 Review） | 编写 `code-review.command` | Command + SOP 文档 |
| W12 | Phase 1 总结、度量对比、PDCA 回顾 | 生成度量报告 | 总结报告 |

**Phase 1 量化目标**：
- 🎯 lint 问题密度降低 40%
- 🎯 单元测试覆盖率从 X% 提升到 X+15%
- 🎯 PB 协议规范符合率达到 80%
- 🎯 Code Review 平均处理时间降低 30%

## 五、核心原则

### 5.1 三层架构

```
┌─────────────────────────────────────────────────┐
│              Layer 3: 度量与反馈                   │
│  量化指标 → PDCA 循环 → 持续改进                   │
├─────────────────────────────────────────────────┤
│              Layer 2: 人机协作工作流                │
│  Rules → Skills → Commands → SOP → CI/CD        │
├─────────────────────────────────────────────────┤
│              Layer 1: 工程素养基座                  │
│  编码规范 → 测试 → 设计 → 架构 → 安全             │
└─────────────────────────────────────────────────┘
```

### 5.2 人机分工原则

| 维度 | 人类负责 | AI 负责 |
|------|---------|---------|
| **需求理解** | 业务判断、优先级决策 | 需求拆解、任务分解 |
| **方案设计** | 架构决策、技术选型 | 方案生成、多方案对比 |
| **编码实现** | 核心逻辑审查、业务逻辑把控 | 代码生成、样板代码、测试用例 |
| **代码审查** | 架构合理性、业务正确性 | 规范检查、安全扫描、性能建议 |
| **质量保障** | 验收标准制定、最终决策 | 自动化测试、覆盖率分析、回归测试 |
| **知识沉淀** | 经验总结、规则制定 | 规则执行、模式识别、文档生成 |

  │            ┌────┘          ┌────┘          ┌──── Phase 1: 基础辅助
  │       ┌────┘          ┌────┘               │    AI 补全 + 人类主导

## 六、风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|-------|------|---------|
| 团队抵触，不愿改变习惯 | 高 | 高 | 从试点团队开始，用数据说话，展示提效效果 |
| Rules 写得太死板，AI 执行效果差 | 中 | 中 | 每月 PDCA 迭代，基于实际反馈持续优化 |
| AI 生成代码质量不稳定 | 中 | 高 | 质量门禁 + 人机交叉 Review 兜底 |
| 过度依赖 AI，工程能力退化 | 低 | 高 | 坚持"人给 AI 立规矩"原则，定期技术分享 |
| 管理层只看迭代效率，不支持质量投入 | 高 | 高 | 用 ROI 数据证明：质量投入 → 减少返工 → 净提效 |
